日期：2026-07-10
范围：重设计后的 Desktop / 960px 中屏 / 411px 手机 / TV Mock 交互
结论：首次审计确认 3 个 P1 交互缺陷和 1 个 P2 状态一致性问题；2026-07-11 修复复验已全部通过。[F-ECHO-015]

## 2026-07-11 修复复验

| 缺陷 | 修复结果 | 点击与断言 |
|---|---|---|
| 引导回放停在第 3 步 | 仅在 Modal 从关闭变为打开时重置到欢迎页，不干扰正常的上一步/下一步 | 完成引导、刷新、从设置回放后，欢迎页、下一步按钮和无上一步状态全部正确 |

| 工作区设置焦点竞争 | 等待 Drawer `afterOpenChange` 完成后再滚动并聚焦“添加目录” | 知识库 Modal → 配置目录 → 设置 Drawer 的完整路径稳定聚焦目标按钮 |

更新防线同时下沉到 `runtime.installAppUpdate`，因此即使 UI 状态过期或 bridge 错报，也不能绕过版本比较安装旧版。Ant Design Modal 的 `destroyOnClose` 已统一迁移为 `destroyOnHidden`。

## 验证方式

本轮不是只跑静态检查，而是分三层实际操作：

1. 在本地 Vite 页面使用可见浏览器逐项点击首次引导、状态诊断、会议、知识库、设置、工作台、检查器、输入框与响应式入口。
2. 使用 headed Chromium 执行完整 E2E 与 scenarios，后端、Electron IPC、文件、麦克风和更新操作使用隔离 Mock，不污染真实 EchoDesk 数据。
3. 对失败项单独重跑，并读取 DOM 焦点、`aria-expanded`、横向滚动宽度、控制台和 Playwright trace。

## 首轮确认的问题（现均已修复）

### P1 · 重新查看引导没有回到第一步（已修复）

完成引导后，从“设置 → 重新看一次引导”进入，实际仍停留在第 3 步“准备就绪”。`OnboardingModal` 的 `stepIdx` 只在组件初始化时设为 0，重新打开时没有复位。

### P1 · 960px 检查器打开后无法点击原按钮收起（已修复）

“打开检查器”正常，但打开后的绝对定位 Inspector 使用更高层级覆盖了右侧切换按钮。Playwright click 与真实 DOM 指针点击后，按钮都保持 `aria-expanded=true`。

### P1 · 知识库跳转设置后的键盘焦点不稳定（已修复）

点击“管理知识库 → 配置目录”后，工作区设置区和“添加目录”按钮都可见，但焦点仍停留在背景中的 `workspace-config-btn`。完整 headed E2E 的两个相关用例均失败；标准模式复跑也出现间歇失败。

失败时的截图与 trace 只作为隔离测试运行的临时 artifact，不进入源码；可长期复核的证据是同一回归用例、修复提交与 CI 运行记录。

### P2 · “已是最新”与下载按钮状态冲突（已修复）

检查更新返回“已是最新”，并显示“本机 v0.3.0（公开发布 v0.2.50）”，但“下载最新版本”仍保持可点击，存在误导用户下载旧公开版本的风险。

### P3 · 控制台兼容性提示

- Ant Design `Modal.destroyOnClose` 已迁移为 `destroyOnHidden`。
- 部分静态 `message` 调用不能读取动态主题上下文。

离线页面中的 backend 500、麦克风初始化超时和服务不可达日志属于本轮刻意验证的失败态，不计为新增 UI 崩溃。

## 安全边界

以下高副作用动作没有在真实用户数据或真实系统上执行，而是在 Mock 中点击完成，或只打开确认框后取消：

- 真正删除会议输出、清空真实知识库、移除真实目录、确认重置说话人。
- 真正下载并安装旧版本、安装后重启应用。
- 修改真实模型密钥、调用付费模型、接受系统麦克风权限。
- 用系统应用打开真实用户文件或下载真实诊断包。

这些边界符合“模拟点击验证”的目标，同时避免破坏本机数据。
